Imagine an android's hands taking a large lump of clay and, through a disciplined workflow, shaping it into an exquisite sculpture. Not from a template. Not from parts. From a single, amorphous mass — progressively constrained into something precise, structural, and beautiful.

That's the mental model behind JanumiCode.

A user starts with a high-level prompt — "build a multi-tenant SaaS platform for real property operations." That prompt is the clay. It contains every possible system that could satisfy the intent. The job of JanumiCode is not to *generate* the system. It's to **reveal the one valid system by removing every invalid possibility**.

---

## The Key Insight: Sculptors Remove, Not Add

When Michelangelo was asked how he carved David, the (possibly apocryphal) answer was: "I saw the angel in the marble and carved until I set him free."

Whether or not he said it, the principle is precise: **the sculpture already exists inside the material. The sculptor's job is to remove what doesn't belong.**

This is not just poetic. It maps directly to how JanumiCode works.

A high-level prompt contains a maximal possibility space — every architecture, every technology choice, every domain model, every integration pattern that could plausibly satisfy the intent. Each phase of the governed workflow applies constraints that eliminate invalid configurations until only one remains: a verified, production-grade system.

This is **constraint satisfaction**, not code generation.

## Progressive Constraint Sculpting

We formalize this as a state model. Each stage represents a reduction in entropy — from raw intent to verified system:

<SculptingStages />

Each transition applies a set of constraints:

**S₀ → S₁**: "What are we actually building?" — The prompt is analyzed, domains are identified, requirements emerge. The formless clay gets its first rough shape.

**S₁ → S₂**: "What rules must it obey?" — Invariants, policies, non-functional requirements. These are the constraint fields that govern all downstream decisions. Material that violates them is removed.

**S₂ → S₃**: "What does it look like?" — Components, interfaces, dependencies. The architecture graph emerges not by invention but by elimination — what structures can actually satisfy all constraints simultaneously?

**S₃ → S₄**: "How do we build it?" — Task decomposition, sequencing, bounded work units. The plan is the toolpath — the sequence of cuts that transforms the rough form into the final detail.

**S₄ → S₅**: "Build it." — Code generation, infrastructure provisioning, claim verification. Each claim is a statement about the sculpture's shape — verified against the constraints before the next cut.

**S₅ → S₆**: "Survive the kiln." — Tests, runtime validation, compliance checks, security scans. The kiln is deterministic reality. It doesn't care what the sculptor intended. It only cares what the sculpture actually is.

## Clay Has Resistance

This is where the metaphor becomes technically powerful.

In real sculpting, clay pushes back. Structure collapses under its own weight if unsupported. Some shapes are physically impossible without scaffolding. The kiln cracks pieces that have internal stresses.

Software has exactly the same properties.

<ClayResistance />

An architecture that looks elegant on paper but can't scale under load is a structure that collapses under gravity. A design that passes unit tests but fails in production is a sculpture that cracks in the kiln. An LLM that hallucinates a dependency that doesn't exist is a tool that cuts in the wrong direction.

**Material resistance is not a bug — it's the fundamental constraint that separates valid sculptures from invalid ones.** JanumiCode's verification layer, claim system, and human gates exist specifically to detect and respond to material resistance before it becomes a production failure.

## The Kiln Is the Most Important Stage

This is where most AI systems fail conceptually.

Many AI coding tools stop at "looks like a sculpture." The output appears plausible. The code compiles. The tests that exist pass. But nobody checked whether the structure can survive real-world forces.

The kiln is **deterministic reality**:
- Tests that exercise actual behavior, not mocked approximations
- Runtime validation under realistic load
- Policy enforcement against compliance requirements
- Security checks against real threat models
- Cost constraints against actual infrastructure pricing

JanumiCode's Verifier, human gates, and governance layer are the kiln. They don't care what the Technical Expert proposed or what the Executor generated. They care whether the system actually works, actually complies, and actually survives contact with production.

**You don't just sculpt — you prove the sculpture is physically realizable before it ever exists.**

## Where the Metaphor Breaks

No metaphor is perfect, and intellectual honesty requires naming the limits.

**Sculpture is static.** A finished sculpture doesn't change. Software systems are living systems — they evolve, adapt, and must respond to changing requirements. JanumiCode addresses this through continuous re-sculpting under governance: each new dialogue is a new sculpting session on an existing form, with full awareness of prior decisions and constraints.

**Sculpture has one artist.** One vision, one pair of hands. JanumiCode is multi-agent, multi-source-of-truth, with potentially conflicting constraints. The resolution mechanism — authority-tiered memory, human gates, and the Mirror & Menu Protocol — is more like a committee of sculptors with a governance charter than a single artist.

**Sculpture has no version control.** You can't undo a chisel cut in marble. Software has Git, branching, and temporal workflows. JanumiCode's append-only event log and supersession tracking give it something sculptors never had: **reversible transformations with full audit trail**.

## The Deeper Pattern: Constraint Fields

The sculpture metaphor is emotionally intuitive and visually powerful. But the deeper technical pattern it encodes is even more precise:

**Progressive constraint application over an initially high-entropy substrate.**

This aligns with:

- **Constraint satisfaction systems** — initial state is maximal possibility space, each step applies constraints, end state is single valid configuration
- **Type theory and program synthesis** — start with underspecified program, end with fully typed executable system, process is constraint propagation
- **Governed AI workflows** — the Executor is the generative force (adds/removes clay), the Verifier is the constraint enforcer (rejects invalid shapes), the Historian checks for consistency with the sculptor's prior work

What JanumiCode does is not generation. It's not autocomplete. It's not even "AI coding."

It's **governed morphogenesis** — the progressive revelation of a valid system from within a space of possibilities, under the continuous enforcement of constraints that represent the physics of software reality.
