# JAN-CMDPRE-DS-001 — Command Precondition Legality Design

*Design-first, **v0.2.1** (D8/D9 amended from the DR-001 critique: the Decision family closes symmetrically — `ApproveDecision` on a waiver discharges the assurance floor — and the `commitState` class is not one rule) — reconciled against an EXECUTED 3-lens adversarial critique (§10: 3 blockers + 4 majors, all folded in below; both weaknesses the v0.1.0 author self-flagged were confirmed and sharpened). Grounded by a multi-agent read-only workflow (6 lenses → adversarial audit per lens → synthesis), 2026-07-22. Scope: the engine-wide defect by which a command RE-ISSUED against an aggregate already in its target state is ACCEPTED and appends a contradicting fact to an append-only record. The demonstrated **privilege escalation** was closed ahead of this design under the working tag `JAN-NOOP-01` (commit pending) — that increment is **DWP-00** in this series. Awaiting design confirmation before the remainder is built.*

---

## 1. What it is (one line)

Make every state-advancing command's **precondition explicit and enforced at the write site**, so the engine stops accepting commands that change nothing (or change the wrong thing) and stops recording events for changes that did not happen.

## 2. What it is NOT (guardrails)

- **NOT** a ban on same-state transitions. Genuine same-state HOLDS exist and are the DOMINANT case on the busiest write path — `ApplyTacticalChange` declares `ACTIVE → ACTIVE` in its own vocab binding, and `ChangePwuState` holds an axis while advancing an orthogonal one in **24 of the seeded undertaking's 67 workLifecycle dispatches** (and 33/35/57 of its sub-axis dispatches; every `ChangePwuState` in the seed holds at least one axis). A classifier-level `from === to` ban breaks the seed wholesale, plus `transitions.test.ts:74-75` and `binding.test.ts:41`. The rule must *permit a hold that is declared* and refuse one that is *accidental*.
- **NOT** a mechanical projection of the vocab's `drivesFrom`. That field has **no ratified authority** (§3), is **absent for twelve commands**, names the **wrong machine** for the single most dangerous command, is **narrower than the machine for eight**, and mirroring it would **revert two already-shipped hardenings**. It is evidence, not a source of truth.
- **NOT** an idempotency mechanism. The bus already returns `DUPLICATE` for an exact `idempotencyKey` replay (`command-bus.ts:104-116`) and that behaviour is untouched. This design governs a **distinct request** — new `commandId`, new key — which every ratified idempotency rule leaves unaddressed (§3).
- **NOT** a repair of history. Events are ratified immutable (DOC-007 §9.1) and the corpus supplies no retraction, compensating-event, or annotation mechanism. Contradictions already written — including any in the seeded workbench — are a **disclosed residual** (§6 D7), not silently fixed.
- **NOT** an authority change. Nothing here grants, withholds, or re-scopes any actor's authority. It refuses commands that the domain's own state machine and each command's own semantics already imply are inapplicable.

## 3. Canonical + governance grounding

**What the corpus REQUIRES (sufficient to ground this without a new rule):**

- **DOC-002 §27** (Canonical Domain Model:1663-1665): "Commands express requested state changes. **Events record accepted state changes.**"
- **DOC-005 §3.2** (:114-116): "Accepted semantic changes produce immutable domain events."
- **DOC-007 §9.1** (:660-666): "Events are immutable." / "Payload contains **the accepted facts, not the original request**."
- **DOC-002 §27.2** (:1687-1700): every handler must "validate preconditions" — the duty exists; the corpus names no *specific* source-state allowlist.
- **DOC-002 §24.2** (:1435-1441), restated executably as **DOC-009 RPH-BAS-005** (:1674-1688) and **Property P7** (:2340-2342): "An authoritative baseline is immutable." **This is the one place the corpus rules on a self-edge, and it rules against it.**
- **DOC-007 §25.1** (:1836-1854): the fifteen error codes are fixed and already include `RPH_ILLEGAL_STATE_TRANSITION`. **No new code need be minted.**
- **Constitution:431**: "For every affected state machine, tests SHALL cover all changed legal transitions, relevant illegal transitions, guards…" — a hard verification obligation on this work.

Taken together: an event emitted for a change that did not occur is a **false entry in an append-only record**, which §27 and §9.1 forbid by construction. That is the ratified footing. This design needs no new governance rule.

**Where the corpus is SILENT — stated plainly, because it determines what is a sponsor call:**

1. **Non-key re-issue is unaddressed.** Every ratified idempotency rule keys strictly on the idempotency key / commandId — DOC-007 §30 (:2138-2141), §35.7 (:2316-2318), DOC-002 §28.2 (:1721), DOC-009 RPH-PER-002 and Property P6. A re-issue carrying a **new** key falls outside all of them. The corpus neither requires rejecting it nor permits accepting it.
2. **`drivesFrom` has no corpus authority at all.** No document mentions it, the M3 binding table, or any duty to enforce a per-command source allowlist. 59 of 84 vocab entries self-declare *"Do NOT treat this sourceSection as proof the shape is ratified"* — including `ApproveBaseline`, a demonstrated exploit site. Enforcing it as a hard gate would promote authored annotations to governance facts, which is exactly what those disclaimers warn against.
3. **The transition class `NOOP` is implementation vocabulary.** The strings "no-op", "NOOP" and "self-transition" appear **zero times** across the ratified RPH corpus. It is the engine's own invention. (It *is* load-bearing in this branch's unratified design docs, so the repo's reasoning depends on it — just not the corpus.)
4. **Whether a zero-change control action may be recorded at all** is unruled outside §24.2's baseline-specific prohibition.
5. **Retraction of already-written contradicting events** has no mechanism anywhere.

**Governance posture.** Every allowlist authored by this design is `UNRATIFIED-AUTHORED`, derived from each machine's own ratified in-arrows and cited as such. The sponsor rulings recorded in §6 (D3 reject-not-absorb; DWP-00 security-first sequencing) were taken 2026-07-22 as delegated authority. The git-commit + NO-PUSH boundary remains the ratification control.

## 4. Current-state findings (grounded)

- **F-1 The defect surface is CLOSED, not open-ended.** Exactly **four** `checkTransition` call sites exist engine-wide: `kit.ts:424`, `intent.ts:106`, `pwu.ts:696`, `execution.ts:573`. Everything in scope flows through one of them. This is the single most important finding for sizing — the fix is bounded.
- **F-2 A self-loop can never be classified illegal.** `classifyTransition` returns `{klass:'NOOP'}` for `from === to` at `stateMachine.ts:43`, **before** the explicit-illegal table (`:44-45`) and before the legal-arrow table. A machine that *wants* to forbid a self-edge cannot express it.
- **F-3 Two siblings in one 80-line module disagree, and the disagreement is the only protection.** `checkTransition` admits NOOP as legal (`kit.ts:119`); `canTransition` counts **only** `'LEGAL'` (`stateMachine.ts:53-55`). That undocumented split is the sole reason `ApproveDecision`, `GrantWaiver`, `PromoteBaseline` and `ActivateExecutionPlan` are safe today — protection by accident, not design. `assertTransition` (`:62-70`) is NOOP-permissive too but has **zero production callers**.
- **F-4 Effects COMPOUND, they do not merely repeat.** `advanceStatus` runs `mutate` against the **already-mutated persisted state** (`kit.ts:432`, `base` is `nextEnvelope(loaded.state, …)`). Executed: a re-issued `SupersedeAssurancePolicy` produced `tags: ["superseded-by:A", "superseded-by:B", "superseded-by:B"]` — unbounded growth. A re-issued `ReviseIntent` took an intent from semanticVersion 2 → 3 → 4 while never leaving `REVISED`.
- **F-5 Blast radius: 41 call sites on NOOP-permissive paths.** 35 `advanceStatus` (assurance 9, execution 7, governance 7, pwa-authoring 5, decomposition 4, runtime-binding 3), 5 `advanceIntent`, 1 `changePwuState`. Assessed **21 HARMFUL, 11 BENIGN, 2 UNCLEAR**, plus **7 execution PLAN-level sites that no inventory examined** — they sit in the file a prior commit claimed to have hardened, because `advanceStep` only ever covered *step*-level commands.
- **F-6 The primitive has four hand-written copies plus an inline fifth.** `kit.advanceStatus`, `intent.advanceIntent` (same defect), `pwu.advancePwuLifecycle` (**already strict**, via `canTransition` — the strongest in-repo precedent), `execution.advanceStep` (already fixed), and `pwu.changePwuState`'s inline three-machine loop. A kit-only fix reaches **one of five**.
- **F-7 Guards are not protection.** 14 of 35 sites pass a `guard`; only **3** actually block a re-issue (the `canTransition` ones). `CompleteExecutionPlan`'s guard (all steps terminal-success) is trivially still true on an already-COMPLETED plan; `ValidatePwa`'s composition gate re-runs and re-passes. Sizing by "has a guard" undercounts.
- **F-8 The one ratified self-edge prohibition is stripped at codegen.** `m2-transitions.json` declares `Baseline.status AUTHORITATIVE→AUTHORITATIVE` illegal citing §24.2, and `gen-transitions.ts:133` deletes it: `if (i.from === i.to) continue; // self-loop "illegal" is a NOOP`. The generator prevents the corpus's own rule from reaching the tables — in **both** directions, so the fix cannot be expressed as vocab data alone.
- **F-9 No machine has a genuine self-loop.** Verified over the built tables: 27 machines, 202 states, **zero** `from === to` entries in any `transitions` array. Every NOOP in the system is synthetic.
- **F-10 The defect has a SECOND half.** Beyond `from === to`, the machine legalises an arrow into a target from **any** of its sources, while a *command* is usually narrower. Live outside execution steps: `DenyWaiver` can be aimed at an `EFFECTIVE` **non-waiver** decision (executed: ACCEPTED, emitting `WaiverDenied` against an approval decision). A from-===-to-only fix closes half the problem.
- **F-11 EIGHT `commitState` sites have no transition check at all** (`assurance.ts:297/706`, `pwa-authoring.ts:89/149/465/521/612/681`). *(Corrected from nine: `pwu.ts:298` sits inside `advancePwuLifecycle`, whose `canAdvanceWorkLifecycle` guard routes through the LEGAL-only `canTransition` — v0.1.0's F-11 contradicted its own F-6, and would have sent an implementer to harden the repo's strictest write primitive.)* A re-issue there is not a NOOP — it is a second *edit* — but the contradicting-fact consequence is identical and **no from-===-to rule at any layer reaches them**.
- **F-12 Transport retries are already absorbed, so a re-issue IS a distinct request.** `command-bus.ts:104-116` returns `DUPLICATE` on an exact key replay. But **no key producer in the repo derives keys from content** — `workbench.ts:120/130-132`, `broker.ts:794`, `seed-workbench.ts:149`, `reference-undertaking.ts:223` are all monotonic counters. So the dedup layer stops a deliberate identical re-send and **nothing else** — notably not a UI double-submit.
- **F-13 `expectedRevision` is ratified-required and unimplemented.** DOC-002 §28.1 (:1708-1717): "All aggregate mutations require an expected revision" / "never silently overwrite." `kit.ts:90-92` **honours** one when sent; **no producer sends one**, and the code comment defers enforcement as "a separate migration … tracked in the log". A second `ApproveBaseline` carrying a revision would already have been rejected.
- **F-14 History cannot be repaired, and contradictions are invisible.** `replay.ts:144-151` dedups only by `e.seq`, so two contradicting events both survive and Property P6 still passes; `traceability-view.ts:41-43` pushes link edges with no dedup, so duplicate events duplicate graph edges in the read model the demo renders.
- **F-15 Behaviour change reaches outside `packages/`.** The demo's server actions dispatch unconditionally from form posts and `return fail(400, …)` on any non-ACCEPTED status (`pwa/[id]/+page.server.ts:445-455`), so a double-submit that silently succeeds today starts surfacing a 400. And `broker.ts:558` records that every `DefinePwuType` runs inside one `dispatchBatch`, "so if any is rejected NONE commit" — a newly-rejecting command aborts its whole batch.

## 5. Proposed scope — make the precondition explicit, at every write site

A single rule, applied uniformly, expressed at the call site:

> **Every state-advancing command declares the PRECONDITION under which it may be issued, over `(loadedState, payload)`. The write primitive enforces it before the machine is consulted.**

A **source-state allowlist is the common special case** and remains the form most commands take. It is not the general form, and v0.1.0 was wrong to make it so (§10-B1): three cases in this engine cannot be expressed as a set of states at all.

- **Most commands** — a state set. `AuthorizeRuntimeBinding` may be issued from `REQUESTED|PARTIALLY_AUTHORIZED`.
- **`DenyWaiver` / `GrantWaiver`** — the discriminator is `decisionType === 'WAIVER'`, a **non-state field**. `Decision.status` legalises `PROPOSED→SUPERSEDED` *and* `EFFECTIVE→SUPERSEDED`, so ANY state allowlist derived from the machine admits F-10's exploit verbatim: `DenyWaiver` aimed at an EFFECTIVE **approval** decision drives a governance authorization to SUPERSEDED and emits `WaiverDenied` about a non-waiver.
- **`ChangePwuState`** — all four axis targets are payload-derived (`p.newState`, `p.executionState`, `p.assuranceState`, `p.shapeIntegrityState`), and this ONE command drives every arrow on four machines *plus* every legitimate hold on each. The only state set that is correct is the machine's entire state list — a declaration whose only correct value is "everything". Its real precondition is **"at least one axis must differ from current"**.

A genuine hold is expressed **honestly, by a precondition that admits it** — for a state-set command, by including the target in its own set (`ApplyTacticalChange` declares `['ACTIVE']` and drives to `ACTIVE`) — never by omitting the declaration.

Sequenced so that risk retires first:

1. **DWP-00 — the demonstrated exploits (LANDED, ahead of this design).** `requireFrom` added to `advanceStatus` + `advanceIntent`; applied at `AuthorizeRuntimeBinding` / `DenyRuntimeBinding` / `RevokeRuntimeCapability` / `ApproveBaseline` / `CompleteAssuranceAssessment` (×3 branches) / `ReviseIntent` / `PublishPwa`. Regression suite `command-reissue-guard.test.ts`.
2. **Make the declaration MANDATORY** — the type-level change that prevents regression (§6 D5).
3. **Author the remaining allowlists** family by family, from each machine's in-arrows.
4. **Reconcile the kernel** so `checkTransition` and `canTransition` stop disagreeing, and restore the stripped §24.2 edge.
5. **Close the `commitState` class** (F-11) — a different mechanism, its own rule.

**Explicitly out of scope, disclosed:** the `expectedRevision` migration (F-13 — overlapping but distinct; a client that re-reads then re-issues passes a revision check and still emits a contradicting event); retraction of already-written events (§3 silence); projection-level contradiction surfacing (F-14).

## 6. Design forks — rulings

- **D1 — Rule shape: a per-command PRECONDITION over `(loadedState, payload)`; a source-state allowlist is its common special case.** *Alternatives:* (a) reject when the produced state is byte-identical to the loaded state; (b) route all legality through the LEGAL-only kernel; (c) a source-state allowlist as the general form (v0.1.0's ruling). **(a) is rejected as the general rule** because it would have *accepted* the `ApproveBaseline` exploit — the second command genuinely *did* change `approvalDecisionId`, and that change was the harm — but it is retained as the correct precondition for `ChangePwuState`, where it is the only expressible one (D10). **(b) is rejected as the sole mechanism**: it cannot express a declared hold and cannot reach a non-state discriminator. **(c) is rejected as the general form** because it provably cannot fix `DenyWaiver` (§5) — v0.1.0 claimed one mechanism closed both halves of F-10 "at no extra cost", and at F-10's own worked example that was false. The precondition subsumes all three shapes; where a command's precondition IS a state set, it is written as one and reads identically.
- **D2 — Layer: the WRITE PRIMITIVE, not the classifier.** A ban inside `classifyTransition`/`checkTransition` breaks `ApplyTacticalChange`, the `ChangePwuState` hold, the reference-undertaking seed, and two green tests (`transitions.test.ts:74-75`, `binding.test.ts:41`). The precondition is a property of the **command**, not of the machine — different commands drive the same arrow with different preconditions — so it belongs where the command is handled. The classifier keeps reporting `NOOP` because that is a true *description*; what changes is that nothing treats a description as an authorization.

  **Amended (§10-M1):** this rejects a *blanket* `from === to` ban. It does NOT forbid consulting the explicit-illegal table BEFORE the `from === to` shortcut — a strictly narrower reorder that is safe (verified: 27 machines, 202 states, **zero** declared self-edges today; after D6, exactly one). That reorder is REQUIRED for D6 to land, and it belongs to D6's scope, not here.
- **D3 — REJECT, not absorb. (Sponsor-ruled 2026-07-22.)** `RPH_ILLEGAL_STATE_TRANSITION`, per DOC-007 §25.1's existing code. DOC-007 §30's accept-without-effect is specified for the same-**key** duplicate, which the bus already implements; a fresh-key command with a different payload is a different request, and silently discarding it hides a real caller error. Cost accepted: a demo double-submit surfaces a 400 (F-15), and a rejecting command aborts its `dispatchBatch`.
- **D4 — Allowlists are HAND-AUTHORED from each machine's in-arrows.** Not generated from `drivesFrom` (§2, §3-2). Concretely: `drivesFrom` names the wrong machine for `CompleteAssuranceAssessment` (declares `AssuranceAssessment.disposition`; all three handler sites drive `AssuranceAssessment.state`); is absent for twelve commands including three exploitable ones; declares non-state sentinels (`(any)`, `(prior)`, `(initial)`) for 23; and generation would silently revert `StartExecutionStep`'s deliberate narrowing and `PruneExecutionStep`'s deliberate widening. Each authored set cites the machine rows it came from.
- **D5 — The declaration becomes MANDATORY, and it is sequenced LAST.** Making it optional (DWP-00's shape) leaves the defect one omission away at every future call site, with no lint, test or type mechanism to catch it; making it **required** on the args type moves that to compile time, which is the durable value worth having. A genuine hold then reads as a *stated* fact (`ApplyTacticalChange` declares `['ACTIVE']`) rather than a silence.

  **Corrected (§10-B3):** v0.1.0 sequenced this FIRST and claimed it enabled family-by-family authoring. That is impossible. `requireFrom` is an optional property on an inline object literal with no default and no spread at any call site, so flipping it to required is a compile error at **31 sites at once** (advanceStatus: assurance 6/9, decomposition 4/4, execution 7/7, governance 6/7, pwa-authoring 4/5, runtime-binding 0/3; advanceIntent 4/5). The branch stays red until all 31 land — one ~31-site behaviour-changing diff across six handler families, precisely the shape that produced this repo's earlier bad remediation. **Ruling: author the preconditions family by family while the field stays optional, then flip it to required as a pure, zero-behaviour type change once the census reaches zero.** Every increment stays independently reviewable and revertible, and no sentinel is needed — which also preserves D5's original "no escape hatch" property, achieved at the end rather than the start.
- **D6 — Restore the §24.2 baseline self-edge, WITH the classifier reorder it requires.** Determinable from the corpus and strictly separable from the general rule. Honest scope: this fixes **table fidelity, not a live hole** — `canPromoteBaseline` already blocks the runtime case via `canTransition`. The generator's other drop branches are load-bearing and must not be disturbed.

  **Corrected (§10-M1):** restoring the row alone lands RED. `rph-domain/src/transitions.test.ts` iterates every `illegal` row of every machine asserting `classifyTransition(...).klass === 'ILLEGAL_EXPLICIT'`, but `stateMachine.ts:43` returns `NOOP` **before** the illegal lookup at `:44`. So D6's scope is three things together — restore the row, stop the generator stripping declared self-edges, and reorder the classifier to consult `illegal` first — or D6 is dropped entirely as decoration. It must not land as "restore a row" with the failing assertion deleted under time pressure, in the increment whose stated purpose is table fidelity. (v0.1.0's §2 named two tests a classifier change would break and MISSED this one, which is the one that bites.)
- **D7 — Already-written contradictions: DISCLOSE, do not repair.** Events are ratified immutable and the corpus offers no retraction (§3-5). A write-path fix stops new contradictions and does nothing about stored ones. **Required:** an audit of the seeded workbench for pre-existing duplicates (the PWA root may already have been re-pointed), reported not rewritten. Projection-level contradiction surfacing is deferred with disclosure.
- **D8 — The wrong-source half (F-10) is IN scope, costs MORE than one mechanism, and must close the Decision family SYMMETRICALLY.**

  **Amended (DR-001 critique B2):** v0.2.0 named only `DenyWaiver`/`GrantWaiver` and thereby missed the **mirror exploit, which is worse**. `authorizeDecisionEffective` (`rph-domain/src/governance.ts:50-62`) checks transition legality and authority — **never `decisionType`** — while `floor-gate.ts:224-233` honours any Decision whose OBJECT reads `decisionType === 'WAIVER' && status === 'EFFECTIVE'`, without regard to which command made it effective. So **`ApproveDecision` aimed at a PROPOSED WAIVER drives it EFFECTIVE and discharges the assurance floor**, recording `DecisionEffective` where a `WaiverGranted` should be — so every waiver-specific control (the waiver detail naming its policy/criterion, review conditions, expiry) is bypassed and no waiver fact exists to audit. The family is therefore closed in BOTH directions or not at all: `ApproveDecision` requires `decisionType !== 'WAIVER'`; `GrantWaiver`/`DenyWaiver` require `=== 'WAIVER'`; `RevokeDecision` (a bare `advanceStatus`, no guard) is audited for the same question.

  The original point stands too: Leaving it open would repeat the earlier error of declaring a family closed while half of it is untouched. **Corrected (§10-B1):** v0.1.0's "same mechanism, no extra cost" was false. The Decision/Waiver family needs a payload predicate (`decisionType === 'WAIVER'`), not a state set, and is the reason D1 was re-ruled. Any increment that signs this family off on a state set alone ships the demonstrated exploit still live.
- **D9 — The `commitState` sites (F-11) are IN scope, as their own increment. There are EIGHT, not nine (§10-M2) — and they are NOT one rule.**

  **Amended (DR-001 critique B4):** v0.2.0 said "they are edits, not transitions — so they get an explicit precondition rather than a source-state set", implying a single rule. That is wrong at four of the eight. `assurance.ts:706` (`SubmitEvidenceForAssessment`) commits with **no state delta by design** — its own comment says "the received-evidence fact lives on the EVENT" — so a rule of the form "an edit that changes nothing appends no event" REFUSES a legitimate command and takes down the claim→evidence→assessment→decision→baseline chain. Its real defect (the same `evidenceId` submitted twice) is **undecidable from `(state, payload)`**, because the received-evidence list exists only in the event log. Two more (`:521 deletePwa`, `:681 removePwuType`) are DELETIONS whose rule is existence, not change; one (`:149 bumpPwaSemanticVersion`) is a derived write with no command of its own. The class needs a per-site rule table, and — load-bearing for D1 — it forces the question of whether `Precondition.check` receives a **reader** as well as `(state, payload)`. That signature must be settled in the FIRST increment, not discovered in the last. They need a *different* rule — they are edits, not transitions — so they get an explicit precondition rather than a source-state set. Sequenced last so the transition class is finished first, but **not deferred**: a designer scoping from the `from === to` framing alone would believe the problem solved while a whole class is untouched.

- **D10 — `ChangePwuState` gets its own increment and its own rule.** It is the engine's highest-traffic write path and it cannot carry a state set (§5). Its precondition is **"at least one of the four axes must differ from current"** — D1's alternative (a), correct *here* and wrong as the general rule. Note the vacuity trap that makes this urgent: for this command the vacuous declaration (every state) IS the correct one, so no type or lint can flag it — D5's compile-time defence is inert exactly where the defect is live. A re-issue with all four axes equal to current and a contradicting `reasonCode` passes the `previousState` staleness check, passes all three sub-axis `checkTransition` calls as NOOPs, bumps the revision and appends a second contradicting `PwuStateChanged`.

  **Sizing correction (§10-B2):** v0.1.0 said the seed exercises this hold "three times". It exercises it at three *call sites*, which fan out to **24 held workLifecycle dispatches of 67** (and 33/35/57 sub-axis holds) — every `ChangePwuState` dispatch in the seed holds at least one axis. The hold is the dominant case, not an edge case, and any rule here must be validated against the seed before it lands.

## 8. Verification approach

- **Per-site**: a regression test that the exploit is refused, **plus** a positive test that the widest legitimate path still works — `requireFrom` narrower than the machine is the likeliest self-inflicted regression (DWP-00 already carries the `PARTIALLY_AUTHORIZED → AUTHORIZED` case).
- **Constitution:431 obligation**: for every affected machine, cover changed legal transitions, relevant illegal transitions, and guards.
- **Mutation discipline**: for each enforcement point, confirm that reverting it makes a named test fail. The prior remediation's most damning audit finding was fixtures shaped so the defect could not appear; that scepticism applies to these tests too.
- **Seed + demo**: the reference undertaking must drive unchanged (it exercises the `ChangePwuState` hold three times), and the PWA-authoring `dispatchBatch` path must still commit.
- **Adversarial verification after build**, read-only, with the finding that a remediation is the most dangerous kind of commit.

## 10. Self-critique and readiness

**A 3-lens adversarial critique was EXECUTED** (design soundness · grounding fidelity · buildability), read-only against the built engine. **17 findings; 3 blockers + 4 majors**, all folded into §5/§6/§9 above. Both weaknesses the v0.1.0 author self-flagged came back **confirmed and sharpened** — which is the argument for running the critique before the roadmap rather than after the build.

- **B1 (D1/D8/§5) → FIXED.** "One mechanism closes both halves of F-10 at no extra cost" was FALSE at F-10's own worked example. `denyWaiver` (`governance.ts:344-352`) is a bare `advanceStatus` to SUPERSEDED with no guard, and `Decision.status` legalises both `PROPOSED→SUPERSEDED` and `EFFECTIVE→SUPERSEDED` — so the allowlist D4 would author, `['PROPOSED','EFFECTIVE']`, **admits the exploit verbatim**, because the discriminator is `decisionType === 'WAIVER'`, a non-state field.
