**Findings**

1. `P1` The proposal validates the wrong ingress path. The CLI contract in [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:45>) feeds `--intent @filepath` directly into the pipeline and relies on auto-approval, but the current in-process path goes through `ClientLiaisonAgent` with attachments/references and routed user input in [workflowDriver.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/helpers/workflowDriver.ts:69), [workflowDriver.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/helpers/workflowDriver.ts:178), and [workflowDriver.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/helpers/workflowDriver.ts:235). That means file resolution, attachment ingestion, query classification, and decision routing can all be broken while the proposed harness still goes green. The spec needs either a liaison-backed harness mode or an explicit statement that the CLI reuses the same ingress adapter as the sidebar flow.

2. `P1` The proposed phase topology does not match the current canonical phase model. The spec adds sub-phases such as `2.4/2.5`, `3.4/3.5`, `4.4/4.5`, `5.4-5.6`, `8.4/8.5`, and `9.5` in [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:249>), but the actual canonical `SUB_PHASE_NAMES` only defines `2.1-2.3`, `3.1-3.3`, `4.1-4.3`, `5.1-5.3`, `7.1-7.3`, `8.1-8.3`, and `9.1-9.3` in [records.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/lib/types/records.ts:137). Without an explicit migration plan for templates, schemas, UI, and record taxonomy, the harness will either fail against the real engine or force the engine to change just to satisfy a test document. This is a spec/code contract break, not just an implementation detail.

3. `P1` The Hestami oracle is overfit to one interpretation of an ambiguous source document. Assertions like “Vision/Mission missing,” “`user_stories` length >= 40,” and required mentions of DBOS/accounting integration in [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:240>) and [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:253>) are not stable facts of the source. The actual Hestami file has a blank top-level mission section in [Hestami doc](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/test-workspace/specs/hestami-ai-real-property-os(2)/Hestami AI Real Property OS and Platform Product Description.md:3>), but also detailed vision/mission statements and modeled-product appendices in [Hestami doc](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/test-workspace/specs/hestami-ai-real-property-os(2)/Hestami AI Real Property OS and Platform Product Description.md:395>) and [Hestami doc](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/test-workspace/specs/hestami-ai-real-property-os(2)/Hestami AI Real Property OS and Platform Product Description.md:677>), while some integrations are explicitly TBD in [Hestami doc](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/test-workspace/specs/hestami-ai-real-property-os(2)/Hestami AI Real Property OS and Platform Product Description.md:336>). This will push the implementation toward keyword-matching and output-shaping rather than faithful reasoning over the document.

4. `P1` The pass criteria ignore the core governance value of the Governed Stream: lineage. The harness contracts in [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:191>) mainly check artifact presence and field contents, but they do not require correct `derived_from_record_ids`, `source_workflow_run_id`, authority progression, or decision-linked provenance. Those fields are first-class in [records.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/lib/types/records.ts:8), and decisions are explicitly persisted as lineage-bearing records in [decisionRouter.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/lib/orchestrator/decisionRouter.ts:46). A phase could fabricate plausible downstream artifacts disconnected from approved upstream records and still satisfy the current spec. That would produce a “working” harness for a broken JanumiCode.

5. `P2` The proposal is built on a stale view of the repo, so it risks creating a second competing oracle instead of consolidating the first one. The background still says only Phase 0/1 harnessing exists and later phases are stubs in [feature brief](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/feature - test strategy harness.md:46>) and [feature brief](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/feature - test strategy harness.md:143>), but the repo already has `workflowDriver`, phase 0-10 handlers, full-pipeline tests, and prompt probes in [workflowDriver.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/helpers/workflowDriver.ts:1), [fullPipeline.test.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/unit/orchestrator/fullPipeline.test.ts:16), [completePipeline.test.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/unit/orchestrator/completePipeline.test.ts:18), [phase2.probe.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/prompt-probes/phase2.probe.ts:11), and [phase3.probe.ts](/E:/Projects/hestami-ai/JanumiCode/janumicode_v2/src/test/prompt-probes/phase3.probe.ts:12). If the spec lands unchanged, the team will be maintaining two sets of expectations with no stated source of truth.

6. `P2` “Full pipeline success” still does not prove value delivery at the execution boundary. The spec explicitly says Phase 9 is intentionally shallow and Phase 10 only checks for records like `commit_record` and `workflow_run_summary` in [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:321>) and [spec](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/test strategy harness - specification.md:330>). That allows a green run even if generated code does not compile, tests are synthetic, or commit metadata is emitted without meaningful workspace verification. For a harness whose purpose is autonomous self-verification, artifact existence is too weak a completion condition.

The common thread is oracle fidelity. The proposal is strong on structure, but it is still too easy for it to certify a simplified, partially invented JanumiCode instead of the real product path, the real phase model, and the real governed-stream guarantees.

I would add one more requirement immediately: version-lock the Hestami corpus itself. The brief says the file is 709 lines in [feature brief](</E:/Projects/hestami-ai/JanumiCode/janumicode_v2/docs/test strategy harness/feature - test strategy harness.md:118>), but the file on disk is currently 708 lines, which is enough to show the input corpus is already drifting.
