## REVIEW TYPE — Semantic Reference Drift (SRD-1) audit  `[ ]` OPEN
**Origin (2026-07-05).** The P1.3c journey→domain remediation arc (fix#3/5/6/7) resolves *lexical* id-drift (separator / case / plural / truncation) via oracle-anchored STRING resolvers, and deliberately does NOT use semantic (embedding) similarity — a blocking integrity gate must fail-SAFE (abstain + flag genuine dangling refs), whereas embedding nearest-neighbor is fail-OPEN (always returns a plausible match, papering over real errors). During that discussion the claim "the drift is empirically LEXICAL, not semantic" was asserted — but it rests on **survivorship bias** and has never actually been checked. This review type exists to check it.

**The gap — the stop-and-fix loop only ever sees drift that trips a BLOCKING validator.** So we have NO evidence about drift that does NOT trip one:
1. **SEMANTIC drift** — the model emits a *synonym/paraphrase* of the canonical id (`DOM-MESSAGING` for `DOM-COMMUNICATION`; `comp-auth-svc` for `comp-user-authentication-service`) at LOW lexical similarity. The lexical resolver correctly ABSTAINS → the ref is kept verbatim → if the join has a validator it's (mis-)flagged as a hallucination; if it does NOT, it is **silently accepted / falls open**.
2. **UNGUARDED joins** — many P2–P8 id-joins have NO referential-integrity validator (unlike P1.3c). Semantic drift there is invisible: it neither trips a gate nor gets resolved.
3. **SILENT MIS-RESOLUTION** — a lexical resolver (token-subset, longest-prefix, ≥0.90 similarity) could resolve a semantic-drift ref to a WRONG-but-lexically-plausible oracle member, and nothing verifies the resolution was semantically correct.

**What the review checks (per id-join site, over a real run DB).** For each non-exact-match ref: classify LEXICAL-resolvable / SEMANTIC-drift-candidate / genuine-hallucination via **dual similarity** — `lexical_sim(ref, nearest_oracle)` LOW (below the resolver threshold) BUT `semantic_sim(ref, nearest_oracle)` HIGH (embedding cosine in a band) ⇒ a synonym/paraphrase the lexical resolvers structurally miss. Also record: (a) the join's VALIDATOR STATUS — gated fail-safe vs falls-open-silently; (b) whether a lexical resolver MIS-resolved (resolved to a member that is NOT the semantic-nearest).

**Scope — id-join inventory to sweep** (LLM-emitted id → oracle it must match; note validator presence): journey→domain (P1.3a ✓), workflow→domain (P1.3b ✓), FR→journey (P2.1c ✓), NFR→SR, component→responsibility/domain (P4), task→component (P6), task→AC (P6/P7), entity→component (P5=PA-4), test→AC (P7), ADR→component (P4), active_constraints→TECH registry (P5/6=PA-9), packet cross-refs (P8).

**Deliverable:** a per-join-site table {has-referential-validator?, lexical-resolvable, SEMANTIC-drift candidates (lexical-low/semantic-high), silent-acceptance, mis-resolution}. Confirms or REFUTES "drift is lexical," and localizes any unguarded semantic drift.

**Remediation direction (only if the audit finds real semantic drift):** a semantic reference-drift **DETECTOR** that surfaces low-lexical/high-semantic refs as **ADVISORY findings for review** — NOT an auto-resolver behind a blocking gate (per the fail-safe argument; embedding nearest-neighbor must never silently rewrite a ref). Plus add referential-integrity validators to the currently-UNGUARDED joins (the larger structural gap).

**Runnable in two passes (GPU-aware):** *Pass 1 (LEXICAL, no GPU — anytime):* enumerate joins + find the lexically-UNRESOLVABLE refs per site (= the semantic-drift OR hallucination candidates) + record validator status. *Pass 2 (SEMANTIC, needs the embedding model — DEFER while a cal run holds the GPU, per the cal-22b VRAM-contention lesson):* embed the Pass-1 candidates + nearest oracle, split semantic-drift (cosine high) from hallucination (cosine low). Reuse the existing `createEmbeddingClient`/`findNearestAbove`. Script home: `scripts/prompt-audit/`.

**Status:** `[ ]` OPEN. Pass 1 is doable now against cal-38/cal-39 (no GPU); Pass 2 after a cal run frees the embedding model.
