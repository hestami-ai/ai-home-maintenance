# **The "Mirror & Menu" Protocol (MMP)**

This framework operates on one core rule: **The Human never writes the spec; the Human only judges the spec.**

The process forces the agents to "hallucinate structure" early, presenting it to the human to correct, rather than waiting for the human to generate it.

### **Phase 1: The Context Mirror (Addressing Poor Articulation)**

*Goal: Turn vague human intent into a falsifiable Model of the World.*

The human starts with a high-level, likely vague intent (e.g., "Add user auth"). Instead of asking open-ended questions ("How do you want auth to work?"), the **Technical Expert** creates a **Context Mirror**.

1. **The "Steel Man" Projection:** The Technical Expert generates a detailed summary of what it *thinks* the human wants, filling in the blanks with industry standards.  
2. **The Anti-Goal Extraction:** The agent explicitly lists what it assumes is *not* happening (e.g., "I assume we are NOT supporting 2FA for this slice").  
3. **Human Action (Judgment):** The human only needs to say "Yes" or "No, actually we do need 2FA."

**Why this works:** It is easier for a human to correct a wrong statement than to generate a right one from scratch.

### **Phase 2: The Decision Menu (Leveraging Human Judgment)**

*Goal: Extract hidden assumptions by forcing trade-offs.*

Hidden assumptions usually hide behind "obvious" decisions. The **Executor** (acting as Architect) must surface these by presenting a **Decision Menu**.

1. **Option Generation:** The Executor proposes 2–3 viable distinct paths (e.g., "Option A: Clerk.js (Fast/Paid)", "Option B: NextAuth (Custom/Free)", "Option C: Rolling our own (Secure/Slow)").  
2. **Trade-off Articulation:** For each option, the agent articulates the *Consequences* and *Required Assumptions* (e.g., "Option A assumes you have a budget for SaaS").  
3. **Human Action (Judgment):** The human selects an option.

**Why this works:** By forcing a choice, you force the human to exercise judgment (their superpower). The choice reveals the hidden constraint (e.g., picking Option B reveals "Cost" is a constraint).

### **Phase 3: The Pre-Mortem (Aligning Expectations)**

*Goal: Validate the "Model of the World" before coding.*

Before a single line of code is written, the **Verifier** runs a simulation of failure.

1. **Assumption Attack:** The Verifier lists the implicit assumptions required for the chosen path to succeed (e.g., "This assumes the existing database schema is strictly normalized").  
2. **Failure Scenario:** The Verifier presents a scenario: "If we proceed, this will break the reporting module because X."  
3. **Human Action (Judgment):** The human validates the risk: "Accept Risk" or "Abort."

### **1\. The "Mirror" Component (Reflective Inquiry)**

This is based on **Active Listening** and **Requirements Validation** techniques.

* **Concept:** Instead of asking "What do you want?", the agent states "Here is what I think you want; correct me."  
* **Scientific Basis:** In cognitive psychology, it is easier for humans to **recognize** errors (Verification) than to **recall** or **generate** complex specifications from scratch (Generation).

This is based on **Decision Theory** and **Choice Architecture**.

* **Concept:** Instead of asking open-ended questions ("How should we build auth?"), the agent provides distinct options with trade-offs ("Option A: Speed, Option B: Security").  
* **Scientific Basis:** This reduces **Cognitive Load** (specifically *decision fatigue*). Humans are excellent at **comparative judgment** (A vs. B) but poor at **blank-slate design**.  
* **Industry Equivalent:** **Architecture Decision Records (ADRs)**, where choices are presented with Context, Options, and Consequences.
