# Governed Stream UI: Multi-Pass Code Review

This document contains the findings of a structured multi-pass code review of the Governed Stream UI and its associated components. 

The review executes targeted passes evaluating the code along 16 distinct dimensions.

---

## Pass 1: System Invariants

An analysis of the `GovernedStreamPanel.ts` and `components.ts` reveals several critical system invariants that must hold true for the UI to remain functional.

1.  **Server-Rendered ↔ Client-Side Contract**: The physical DOM structure generated by the Extension Host server (`components.ts`) serves as a strict API contract for the client-side JavaScript (`main.ts`, `mmp.ts`). Client handlers hard-rely on `data-action` attributes and exact structural paths (e.g., using `scopedQuery(cardId, ...)`). Any mismatch gracefully silences functionality rather than explicitly throwing errors.
2.  **State Trichotomy Alignment**: The system relies on synchronized consistency across three distinct storage domains:
    *   **Extension Host In-Memory**: Tracking active processes (`_activeDialogueId`, `_isProcessing`, `_activeCLICommandId`, EventBus AbortControllers).
    *   **SQLite Database**: Source of truth for workflow state, logs, and events.
    *   **Webview Client State**: Local UI state, including MMP decisions and clarification threads. 
    *   **Invariant**: The Webview Client State must survive and automatically re-hydrate after full DOM re-renders orchestrated by the Extension Host.

## Pass 2: Failure Modes

The primary failure modes of the Governed Stream UI emerge from violating the invariants identified in Pass 1. 

1.  **Destructive Re-renders**: Triggering `this._update()` in `GovernedStreamPanel.ts` executes a full DOM replacement. Any ephemeral state in the Webview that isn’t carefully serialized and caught during the `postMessage` pipeline is irrevocably lost (e.g., `state.gateRationales`). This creates race conditions where incoming backend events can wipe out human input halfway through typing.
2.  **Selector Drift and Silent Failures**: Updating HTML markup in `components.ts` without corresponding updates in `main.ts` or `mmp.ts` breaks selectors. Because interaction is delegated via global click listeners matching `data-action`, broken selectors fail silently. For example, `.verdict-badge` relies on in-place DOM updates via `handleClaimUpdated()`.
3.  **Dangling Processing States**: The system maintains two separate UI representations of "Processing" (`.processing-cancel-bar` encoded in the host HTML, and `#processing-indicator` manipulated by client JS). If workflow cancellation or gate triggering throws an unhandled exception before cleaning both up, the UI locks in a permanently loading state.
4.  **Regex Injection Brittleness**: The `wrapResizable()` utility relies on regex to inject inline styling into the first element of an HTML payload. This is intrinsically brittle and fails if the structure changes (e.g., an HTML comment is placed at the top of the template).

## Pass 3: Operational Consequences

Analyzing `dataAggregator.ts` reveals severe operational scaling bottlenecks inherent in the aggregation design:

1.  **Linear CPU Degradation on Stream Length**: The `buildStreamItems()` function reconstructs the entire sequence of `StreamItem` objects from scratch every time it is called. For long dialogues, mapping DB events (`dialogue_events`, `gates`, `verdicts`, `claims`) requires extensive repeated iteration.
2.  **Synchronous JSON Parsing Overhead**: During `buildStreamItems()`, massive strings in `event.detail` and `event.content` are synchronously parsed using `JSON.parse()` for almost every intake and architecture event. As the timeline grows, this introduces meaningful synchronous blocking on the Node JS event loop.

## Pass 4: Coupling and Long-Term Maintainability

1.  **High Component Coupling**: To add a single new card type to the Stream, a developer must touch:
    *   The `StreamItem` union type definition.
    *   The `dataAggregator.ts` if/else parsing chain (spanning ~700 lines).
    *   The `components.ts` central switch statement and specific markup generators.
    *   The Client webview handlers in `main.ts` or `mmp.ts` for interactivity.
2.  **Monolithic God Classes/Files**: `GovernedStreamPanel.ts` (3,400+ LOC), `components.ts` (5,200+ LOC) and `styles.ts` (6,400+ LOC) act as catch-all boundaries. Maintaining or reviewing changes within these files carries immense cognitive load and heightens the risk of regression. There is essentially zero module boundary isolation between structural UI logic and workflow domain logic.

## Pass 5: Requirements Mismatch

1.  **Prompt Instruction Contradictions**: The Proposer prompts explicitly assert "Do NOT use markdown code fences", yet intrinsically include code fences within their own schema examples. This places the LLM in a paradoxical constraint scenario, which degrades output reliability.
2.  **Inconsistent Tool Constraints**: The Executor Unit prompt effectively constrains arbitrary tool-use, whereas the Architecture sub-phase prompts omit these safety constraints. 
3.  **Provider Routing Ambiguity**: The Context Engineer specifically requires Context Engineering behavior, but it routes through the `TECHNICAL_EXPERT` LLM provider wrapper. This risks assigning the wrong underlying model or temperature setting to the Context Engine, mismatching capability requirements against the active model profile.

## Pass 6: Data Integrity

1.  **Data Loss on DOM Swap**: Because Gate Rationales (`state.gateRationales`) are held solely in the webview memory space, an asynchronous event (like a `dialogue:turn_added`) triggering `this._update()` will flush the DOM and wipe out the user's unfinished text input. The state re-hydration mechanisms (`_postClarificationThreads`, `_postPendingMmpDecisions`) do not uniformly protect all inputs.
2.  **Weak Typed SerDes**: `dataAggregator.ts` pulls `event.detail` and `event.content` from SQLite and blindly calls `JSON.parse()`. A poorly formed string from an LLM hallucination will crash the aggregator loop, corrupting the entire timeline view and causing the visual stream to fail rendering for the human operator.
3.  **Process-State Orphanage**: The Extension Host tracks `_activeCLICommandId`. If VS Code restarts mid-execution, the SQLite database may record a dangling process state, but the memory pointer is destroyed. The stream may indefinitely wait for a `cli:activity` 'complete' event that will never emit.

## Pass 7: Security Boundaries

1.  **Webview HTML Injection Vector**: The security boundary between the Extension Host and the Webview Client is heavily blurred. The Host generates unvalidated raw HTML templates in `components.ts` and transmits them to the Webview via `postMessage({ type: 'turnAdded', data: { html } })`, where the Webview likely assigns it to `innerHTML`. While `escapeHtml()` is used, any lapse in applying it to LLM-generated output or user commands creates a direct Cross-Site Scripting (XSS) vulnerability within the elevated VS Code editor context.
2.  **Inadequate Tool Invocation Boundaries**: The Architecture Expert prompts lack explicit constraints against arbitrary tool use. Given the agent's elevated access in the workflow, a compromised or halllucinating LLM could invoke CLI tools unexpectedly, executing unapproved system commands outside the intended architectural scope.

## Pass 8: Rollout/Migration Risks

1.  **Tight HTML-JS Schema Coupling**: Because the client Webview JS queries the DOM using specific structural characteristics (e.g., `#processing-indicator`, `data-action`), migrating or upgrading the UI requires synchronized atomic deployments of both `components.ts` and the compiled webview scripts (`main.ts`, `commandBlocks.ts`). Rolling out a backend upgrade without a corresponding client refresh will softly soft-brick the interface.
2.  **SQLite Schema Rigidity**: The `dataAggregator.ts` hardcodes its parsing logic against the current `dialogue_events` and metadata schemas. If a DB schema migration occurs in future rollouts, the naive `JSON.parse` loops will throw type errors or fail to aggregate historical dialogue instances correctly.

## Pass 9: Correctness

1.  **Prompt Instruction Contradictions**: As noted, Proposer prompts instruct agents to avoid Markdown code fences while presenting code fences in their output templates. This fundamentally reduces the correctness of LLM parser ingestion.
2.  **Dual Indicator Discrepancy**: The system maintains `.processing-cancel-bar` (Server) and `#processing-indicator` (Client). A correct system would unify state representation. Here, a failure during gate triggering clears the Extension Host processing state but can leave the frontend JS spinner orphaned, displaying an incorrect visual state to the user.

## Pass 10: Reliability

1.  **Regex Injection in `wrapResizable()`**: The `components.ts` utility `wrapResizable()` uses string-level regex replacement to inject inline styles into HTML strings. This is highly fragile; if an HTML comment is prepended or attributes are re-ordered in future logic, the regex fails, and the UI component reliably silently loses resize capabilities.
2.  **Fallback Pathing Flaws**: Evaluators fail-open to `PROCEED` upon provider failure. While this prevents blocking, it reduces system reliability by silently allowing potentially unverified or hallucinated proposals to bypass quality gates during an API outage.

## Pass 11: Security

1.  **Cross-Site Scripting (XSS)**: By continuously taking LLM output (`turn.content`) and piping it through `components.ts` string concatenation into a Webview via `postMessage`, any missed `escapeHtml` call directly turns the LLM into an XSS vector. While a rudimentary `escapeHtml` is present, it is repeatedly applied manually rather than via a secure templating engine (like React/DOMPurify), making human error inevitable.
2.  **Unconstrained Tool Access**: The `EXECUTOR_UNIT` specifies tight guardrails against arbitrary changes, but Architecture agents omit these constraints. An attacker (or a hallucinating LLM) leveraging the Architecture sub-phase could execute malicious commands on the host machine.

## Pass 12: Performance

1.  **O(N²) Rendering Bottleneck**: For every new `dialogue:turn_added` event or character streamed, `GovernedStreamPanel.ts` calls `_update()`. This triggers `dataAggregator.ts` to iterate over all DB events (O(N)), followed by `components.ts` regenerating a massive monolithic HTML string for the entire conversation history, which the Webview parses via `innerHTML` swap. As conversations grow, this quadratic DOM destruction/re-creation will cause severe browser layout thrashing, UI stuttering, and high memory spikes.
2.  **Synchronous God-Loop**: The `renderStream` function handles 28 complex switch cases synchronously without yielding to the event loop, causing Extension Host unresponsiveness during heavy render cycles.

## Pass 13: Maintainability

1.  **God Objects**: `GovernedStreamPanel.ts` (3,426 lines) and `components.ts` (5,273 lines) are textbook "God Objects". They own too many responsibilities (messaging, event bus subscription, DOM generation, workflow orchestration). This violates the Single Responsibility Principle and creates a massive maintenance burden for onboarding new developers.
2.  **Shotgun Surgery Anti-Pattern**: Adding a single new interactive visual card requires modifying `StreamItem` union types, the aggregator mapping in `dataAggregator.ts`, the switch statement and HTML builder in `components.ts`, the CSS literal in `styles.ts`, and the webview message router in `main.ts`.

## Pass 14: Architecture Compliance

1.  **Violates Separation of Concerns (MVC Boundary)**: Based on modern architecture standards, the Extension Host (Backend Context) should serve structured JSON state arrays to the Webview (Frontend Context). By hard-rendering raw HTML strings in the Host and executing un-escaped `innerHTML` swaps in the client, the architecture fundamentally fails to establish a proper API boundary. It effectively implements server-side rendering (SSR) for a thick-client architecture without a shadow DOM.

## Pass 15: Testing Adequacy

1.  **Testability Black Hole**: Logic deeply nested inside `components.ts` HTML literal templates and 240-line `_handleSubmitInput()` functions cannot be easily strictly unit-tested. 
2.  **Implicit Database Mocking**: Because `dataAggregator.ts` relies heavily on invoking global DB fetching singletons, any unit tests verifying stream generation would require complex, brittle database mocking rather than pure data-in/data-out testing.

## Pass 16: Language Idiom

1.  **String Concatenation vs. Declarative UI**: Writing 5,200 lines of `"<div class=\"foo\">" + var + "</div>"` in TypeScript is highly non-idiomatic for modern extension development. TypeScript idiom for UI heavily favors templated declarative approaches (e.g., TSX, React, or lit-html) which inherently provide XSS protection and type-checked DOM hierarchies.
2.  **Weak TypeScript Strictness**: The continual reliance on `JSON.parse(event.detail) : {}` piped directly into `Record<string, unknown>` interfaces effectively opts-out of TypeScript's compiler safety net, casting arbitrary string dictionaries to workflow interfaces at runtime without Zod/Joi schema validation.
