# JanumiCode Pipeline Design — Intent to Implementation

**Status:** Draft for review
**Scope:** The full transformational chain from underspecified user prompt to verified, built code, for the *product/feature engineer profile*. Other profiles (platform engineer, scientist) are out of scope for this document and will be addressed later.

---

## 1. Purpose of this document

This document specifies the end-to-end transformational pipeline JanumiCode performs to take a human's underspecified product/feature idea and deliver verified, built code. It is intended as a single anchor that can be read in one sitting, marked up, and used as the contract for downstream prompt template rewrites and the vocabulary rename.

It exists because the conversation that produced it was forced to evaluate phases in isolation, which was insufficient for confident decisions. This document presents the chain end-to-end so that each phase can be evaluated *in context of its neighbors*.

It is not yet a code change. It is the design that, once approved, drives the code changes.

---

## 2. The four levels of abstraction

The pipeline is organized into four levels. Each level answers a distinct question about the product. Each level produces artifacts that the next level consumes. Each transition between levels is a *verifiable state transition*: the upstream artifact is a minimal, schema-rigid, deterministically validatable input to the next agent.

| Level | Question | Primary artifact |
|---|---|---|
| **L1 — Intent** | What does the user want, and for whom? | Stakeholder needs |
| **L2 — Experience** | How does the user actually interact with what gets built? | User journeys |
| **L3 — Structure** | How is the system internally organized? | Bounded contexts + capability tree |
| **L4 — Realization** | What gets built, and how is it verified? | Components, tasks, code, tests |

L4 has internal structure because it is recursive in two dimensions:
- **L4a — Architecture decomposition**: components and interfaces, recursively decomposed (RAD).
- **L4b — Task decomposition**: implementation tasks under each leaf component, recursively decomposed.
- **L4-EXEC — Execution**: the agent that writes source code and tests at each leaf task.

Two cross-cutting verification concerns run alongside the level transformations, and they serve completely different purposes:

- **Concern A — Phase-transition validators**: deterministic checks at every phase boundary that confirm intent fidelity is preserved. These do not test built code; they validate artifacts before they are accepted as the next phase's input. They are not a phase. They are a discipline applied to every arrow in the diagram.
- **Concern B — V-model built-system tests**: test code generated by L4b/L4-EXEC alongside implementation code. These are unit tests, component tests, integration tests, and acceptance tests that run *against the built system* to confirm what was built matches what was specified. They are recursive in the same shape as the L4a and L4b decomposition trees.

Both concerns are required. Without Concern A, the pipeline produces internally consistent artifacts that may not match user intent. Without Concern B, built code may pass internal validation but fail to satisfy real-world behavior. Together, they provide end-to-end fidelity from intent through implementation.

## 3. The pipeline at a glance

```
┌─────────────────────────────────────────────────────────────────────┐
│  User prompt (underspecified)                                       │
└────────────────────────────────┬────────────────────────────────────┘
                                 │
                          ┌──────▼──────┐
                          │  L1 Intent  │
                          └──────┬──────┘
                                 │ stakeholder_needs[]
                          ┌──────▼──────────┐
                          │  L2 Experience  │
                          └──────┬──────────┘
                                 │ journeys[]
                          ┌──────▼─────────┐
                          │  L3 Structure  │
                          └──────┬─────────┘
                                 │ bounded_contexts[], capability_tree
                          ┌──────▼──────────────┐
                          │  L4a Architecture   │  ◄──┐
                          │  (recursive RAD)    │     │ recurses
                          └──────┬──────────────┘  ───┘
                                 │ leaf_components[]
                          ┌──────▼──────────────┐
                          │  L4b Tasks          │  ◄──┐
                          │  (recursive)        │     │ recurses
                          └──────┬──────────────┘  ───┘
                                 │ leaf_tasks[]
                          ┌──────▼──────────────┐
                          │  L4-EXEC Execution  │
                          │  (code + tests)     │
                          └──────┬──────────────┘
                                 │
                          ┌──────▼──────────────┐
                          │ Built system + V-model tests
                          └─────────────────────┘

Concern A (phase-transition validators) runs at every arrow.
Concern B (V-model tests) is generated parallel to L4b and runs against built system.
```

## 4. Bloom-and-prune at every phase

Every phase that involves an LLM agent uses the same interaction pattern:

1. **Bloom**: the agent produces a *generous superset* of what the user might mean, drawing on its broad training to expand the input. This is the "bloom" half — over-propose, don't under-propose.
2. **Prune**: the user reviews the proposals via Mirror & Menu and marks them accept / reject / edit / defer. The user is never asked to compose; they are only asked to choose.
3. **Lock**: the pruned set becomes the verified state for the next phase.

The user never sees framework names. They see plain-English questions like "Who will use this?" or "What are the main parts of the system?" The framework precision is internal to the agent prompts, not user-facing.

## 5. Phase-by-phase specifications

For each phase below: the agent's job, the input it receives, the output schema, the deterministic validators that gate the phase boundary, and (where useful) a worked example using the Hestami prompt as input.

### 5.1 L1 — Intent

**Job**: Take a free-form user prompt and produce a structured, prunable description of *what is to be built and who it serves*.

**Input**: Free-form user text. May be a single sentence, a paragraph, or a multi-page document like the Hestami product description.

**Internal framework**: JTBD-flavored stakeholder needs.

**Output schema**:

```ts
L1_Output {
  vision_statement: string                    // free text, max 500 chars, single statement
  product_shape: {                             // structural hint about what's being built
    kind: "single-feature" | "multi-feature-product" | "multi-pillar-platform" | "tooling"
    pillars: string[]                          // empty unless kind == multi-pillar-platform
    notes: string                              // free text, max 500 chars
  }
  stakeholders[]: {
    stakeholder_id: string                    // "STK-001"
    name: string                              // "Homeowner", "HOA Board Member", "Field Technician"
    role: string                              // free text, max 200 chars
    context: string                           // free text, max 500 chars, why they matter
  }
  stakeholder_needs[]: {
    need_id: string                           // "NEED-001"
    stakeholder_id: string                    // FK to STK-*
    situation: string                         // free text, max 300 chars, when they need this
    motivation: string                        // free text, max 300 chars, what they want
    expected_outcome: string                  // free text, max 300 chars, what success looks like
  }
  out_of_scope[]: {                           // bloomed but pruned items
    item_id: string                           // "OOS-001"
    statement: string                         // free text
    reason: string                            // why it was pruned (user-supplied or LLM-inferred)
  }
}
```

**Concern A validators at L1 → L2 boundary**:

1. `vision_statement` is non-empty and within length bounds.
2. `product_shape.kind` is one of the four enum values.
3. Every `stakeholder_id` is unique within `stakeholders[]`.
4. Every `need.stakeholder_id` resolves to an existing `STK-*`.
5. Every need has all three of `situation`, `motivation`, `expected_outcome` non-empty.
6. At least one stakeholder exists and at least one need exists.

**What L2 actually consumes**: only `stakeholder_needs[]`. The vision_statement, product_shape, and stakeholders[] are *contextual* and shipped alongside but are not the verified state that L2 transforms. They remain available as background context to the L2 agent prompt but the validator at the L2 boundary only requires that needs are well-formed.

**Worked example using the Hestami prompt**:

After bloom-and-prune, L1 might produce (abbreviated):

```
vision_statement: "An integrated, holistic, end-to-end home services AI operating system
                   for property owners, community associations, and service providers."
product_shape: {
  kind: "multi-pillar-platform"
  pillars: ["Hestami Home Real Property Assistant",
            "Hestami Community Association Management",
            "Hestami Service Provider Field Services Management"]
  notes: "Three pillars share a platform but serve distinct customer segments."
}
stakeholders: [
  { STK-001, "Individual Homeowner", "Owns one residence", "..." }
  { STK-002, "Small Landlord", "Owns a few rental properties", "..." }
  { STK-003, "Property Investor", "Manages multiple residences", "..." }
  { STK-004, "HOA Board Member", "Volunteer governance role", "..." }
  { STK-005, "Community Association Manager (CAM)", "Professional manager", "..." }
  { STK-006, "Service Provider Owner", "Runs a contracting business", "..." }
  { STK-007, "Field Technician", "Performs the work on-site", "..." }
  ... (bloomed; user prunes which to keep)
]
stakeholder_needs: [
  { NEED-001, STK-001, "When my plumbing fails",
                       "I want to find a reliable contractor quickly",
                       "So that the issue is resolved without me wasting time vetting strangers" }
  { NEED-002, STK-001, "When I receive multiple bids for the same job",
                       "I want to compare them on equal terms",
                       "So that I can pick the best value, not just the lowest price" }
  { NEED-003, STK-005, "When the HOA board meets to approve assessments",
                       "I want auditable financial records to present",
                       "So that the board can vote with confidence and the result survives legal challenge" }
  ... (15-25 needs after bloom, fewer after prune)
]
out_of_scope: [
  { OOS-001, "Real estate brokerage / property listing", "User pruned: Hestami is post-purchase only" }
  { OOS-002, "Mortgage origination", "User pruned: Hestami is post-purchase only" }
]
```

The user never sees the JSON. They see Mirror cards for each stakeholder and each need, and Menu choices for the product_shape. They prune until the set reflects what they actually meant.

### 5.2 L2 — Experience

**Job**: Take the L1 stakeholder needs and produce concrete user journeys describing how the user actually interacts with the system to satisfy each need.

**Input**: `stakeholder_needs[]` from L1. The vision/product_shape/stakeholders are available as context but not consumed structurally.

**Internal framework**: User Journey Maps. Pure journeys, no service-blueprint backstage. Anything not user-visible is L3's job, not L2's.

**Output schema**:

```ts
L2_Output {
  journeys[]: {
    journey_id: string                              // "JRN-001"
    source_stakeholder_need_ids: string[]           // FKs to NEED-* (one journey may serve multiple needs)
    name: string                                    // free text, max 200 chars
    trigger: string                                 // free text, max 300 chars
    outcome: string                                 // free text, max 300 chars
    steps[]: {
      step_id: string                               // "JRN-001-S01"
      sequence_index: integer                       // 1-based, monotonically increasing
      actor: "stakeholder" | "system"               // who acts in this step
      action: string                                // free text, max 300 chars
      inputs: string[]                              // what the actor brings/uses
      outputs: string[]                             // what the step produces
      decision_branches[]?: {                       // optional, only if step forks
        condition: string
        next_step_ids: string[]
      }
      next_step_ids: string[]                       // empty array = terminal step
    }
  }
}
```

**Concern A validators at L2 → L3 boundary**:

1. Every `source_stakeholder_need_ids` entry resolves to an existing `NEED-*` from L1.
2. Every `step_id` is unique within its journey (and globally, by construction since they include the journey ID).
3. Every `next_step_ids` entry resolves to a `step_id` in the same journey.
4. The step graph (sequence + branches) is acyclic.
5. Exactly one step per journey has `sequence_index == 1` (entry).
6. At least one step per journey has empty `next_step_ids` (terminal).
7. Every step has non-empty `action`.
8. `actor` is restricted to the two enum values.
9. **Coverage**: every `NEED-*` from L1 is referenced by at least one journey. Needs that are deliberately deferred land in L1's `out_of_scope[]` instead.

**What L3 consumes**: every journey, every step. The L3 agent's procedural job is to map steps to bounded contexts and capabilities.

**Worked example continuing the Hestami case**:

For NEED-001 ("Find a reliable contractor when plumbing fails"), the L2 agent might produce:

```
journey: {
  JRN-001
  source_stakeholder_need_ids: [NEED-001]
  name: "Find and engage a contractor for an urgent home repair"
  trigger: "Homeowner notices a plumbing leak"
  outcome: "A vetted contractor is scheduled to perform the repair"
  steps: [
    { JRN-001-S01, 1, stakeholder, "Homeowner opens the app and describes the problem",
      inputs: [problem description, photo], outputs: [service request], next: [S02] }
    { JRN-001-S02, 2, system,      "System matches the request to candidate contractors",
      inputs: [service request], outputs: [contractor candidate list], next: [S03] }
    { JRN-001-S03, 3, system,      "System solicits bids from candidates",
      inputs: [candidate list], outputs: [bid set], next: [S04] }
    { JRN-001-S04, 4, stakeholder, "Homeowner reviews normalized bid comparison",
      inputs: [bid set], outputs: [bid selection decision], next: [S05] }
    { JRN-001-S05, 5, system,      "System confirms the chosen contractor and schedules the visit",
      inputs: [bid selection], outputs: [confirmed appointment], next: [] }
  ]
}
```

Notice what is not in this artifact: there is no mention of DPOR license verification, no mention of insurance certificate parsing, no mention of background processes. Those are real things that have to happen, but they are *L3's job to discover*. L2 only describes what the user does and what the system response is.

### 5.3 L3 — Structure

**Job**: Take the L2 journeys and produce (a) a set of bounded contexts that partition the system's vocabulary and ownership, and (b) a capability tree where each capability lives in exactly one bounded context and traces back to journey steps.

**Input**: `journeys[]` from L2.

**Internal framework**: DDD bounded contexts + capability modeling. Every step in every journey will be mapped to a (bounded_context, capability) pair. New bounded contexts and capabilities may also emerge from the system's *unspoken* needs (e.g., "the candidate matching step implies the system needs an entity catalog of contractors, which implies a vendor management context").

**Output schema**:

```ts
L3_Output {
  bounded_contexts[]: {
    bounded_context_id: string                      // "BCTX-PROPERTY"
    name: string                                    // human-readable
    description: string                             // what this context is responsible for
    ubiquitous_language: string[]                   // glossary of nouns/verbs that mean one specific thing in this context
    ownership_rules: string                         // free text describing what this context authoritatively owns
  }
  capability_tree: {
    capabilities[]: {
      capability_id: string                         // "CAP-CONTRACTOR-DISCOVERY"
      bounded_context_id: string                    // FK to BCTX-*; exactly one
      parent_capability_id: string | null           // tree structure
      label: string
      description: string                           // what this capability does
      stakeholder_needs_served: string[]            // FKs to NEED-* from L1
      contributing_journey_step_ids: string[]       // FKs to step_ids from L2
      acceptance_intent: string                     // plain-English statement of "what does it mean for this capability to be present and working"
    }
  }
}
```

**Concern A validators at L3 → L4a boundary**:

1. Every `bounded_context_id` is unique.
2. Every capability's `bounded_context_id` resolves to a known context.
3. Every capability's `parent_capability_id` (when non-null) resolves to another capability *in the same bounded context*. (Capability hierarchies do not cross context boundaries.)
4. The capability tree is acyclic.
5. **Coverage of journeys**: every step from every journey appears in *at least one* capability's `contributing_journey_step_ids`. The mapping is total. No orphaned steps.
6. **Coverage of needs**: every `NEED-*` from L1 is referenced by at least one capability's `stakeholder_needs_served`.
7. Every capability has a non-empty `acceptance_intent`.
8. Tree depth does not exceed 4 levels (configurable hint, not a hard requirement, but flagged for review if exceeded).

**The two open questions in this phase**, with my recommendations:

**Open Question 1: How does the L3 agent propose initial bounded contexts?**

- *Path A — Step grouping*: cluster journey steps by language and entity affinity.
- *Path B — Glossary first*: extract vocabulary (nouns/verbs) from journeys, cluster vocabulary, each cluster becomes a candidate context.

**Recommendation: Path B (glossary-first).** It is closer to how Eric Evans defines bounded contexts ("where one ubiquitous language holds"), and vocabulary extraction is more deterministic than judgment-based clustering. The L3 prompt instructs the agent to extract every distinct noun and verb from every step, then propose contexts where each noun/verb has exactly one meaning.

**Open Question 2: How does the L3 agent name and scope capabilities within a context?**

- *Path A — Action-derived*: every journey step → one candidate capability.
- *Path B — Need-derived*: capabilities cluster around stakeholder needs, not steps.
- *Path C — Hybrid*: capabilities are need-derived for naming and granularity, *and* every capability carries an explicit list of contributing journey step IDs for traceability.

**Recommendation: Path C (hybrid).** Need-derived gives the right granularity (a capability is "an ability to satisfy a kind of need," not "a wrapper for a single step"). Step ID linking gives deterministic coverage validation. The schema above already reflects this hybrid by including both `stakeholder_needs_served` and `contributing_journey_step_ids` on every capability.

**Worked example continuing the Hestami journey above**:

After processing JRN-001, the L3 agent might produce:

```
bounded_contexts: [
  { BCTX-PROPERTY,            "Property",            "Properties, units, addresses, ownership", ... }
  { BCTX-SERVICE-REQUEST,     "Service Request",     "Captures the why and scope of work", ... }
  { BCTX-VENDOR,              "Vendor",              "Contractor catalog and credentials", ... }
  { BCTX-BIDDING,             "Bidding",             "Solicits, normalizes, and presents bids", ... }
  { BCTX-SCHEDULING,          "Scheduling",          "Calendar, dispatch, appointments", ... }
]

capability_tree.capabilities: [
  { CAP-001, BCTX-SERVICE-REQUEST, parent: null, "Capture a service request",
    needs_served: [NEED-001], step_ids: [JRN-001-S01],
    acceptance_intent: "A homeowner can describe a problem with text and a photo,
                        and the system stores it as a structured service request." }

  { CAP-002, BCTX-VENDOR,          parent: null, "Maintain a catalog of qualified contractors",
    needs_served: [NEED-001], step_ids: [],   // implied need, no direct journey step
    acceptance_intent: "The system has a current list of contractors with verified credentials." }

  { CAP-003, BCTX-VENDOR,          parent: null, "Match contractors to a service request",
    needs_served: [NEED-001], step_ids: [JRN-001-S02],
    acceptance_intent: "Given a service request and the contractor catalog, the system produces
                        an ordered candidate list ranked by fit." }

  { CAP-004, BCTX-BIDDING,         parent: null, "Solicit bids from candidate contractors",
    needs_served: [NEED-001], step_ids: [JRN-001-S03],
    acceptance_intent: "Each candidate contractor is asked to bid, with a deadline,
                        and bids are collected." }

  { CAP-005, BCTX-BIDDING,         parent: null, "Normalize and present bids for comparison",
    needs_served: [NEED-001, NEED-002], step_ids: [JRN-001-S04],
    acceptance_intent: "Bids with different scopes are normalized to comparable units
                        and presented side by side." }

  { CAP-006, BCTX-SCHEDULING,      parent: null, "Confirm a contractor and schedule the visit",
    needs_served: [NEED-001], step_ids: [JRN-001-S05],
    acceptance_intent: "Once selected, the system books an appointment and notifies both parties." }
]
```

Notice how CAP-002 ("Maintain catalog of qualified contractors") emerged *without* a direct journey step. This is the L3 agent doing real work — recognizing that step S02 ("System matches the request to candidate contractors") implicitly requires that *some other capability* supplies the contractor list. The glossary-first approach surfaces "contractor catalog" as a noun used in the journey, which becomes a bounded context (BCTX-VENDOR), which then needs at least one capability that maintains it. The validator ensures every step is covered (S02 → CAP-003), but the agent is allowed to add capabilities that have no step IDs as long as they are needed to make other capabilities operationally meaningful.

### 5.4 L4a — Architecture (recursive)

**Job**: Take the capability tree and produce a tree of components and interfaces such that every leaf capability is supported by exactly one leaf component. Recursively decompose any component flagged as too complex (RAD).

**Input**: `bounded_contexts[]` and `capability_tree` from L3.

**Internal framework**: Component-and-Connector decomposition with RAD.

**Output schema**:

```ts
L4a_Output {
  components[]: {
    component_id: string                            // "COMP-CONTRACTOR-MATCHING"
    parent_component_id: string | null              // RAD parent
    label: string
    responsibility: string
    bounded_context_id: string                      // FK; a component normally serves one context
    supported_capabilities: string[]                // FKs to CAP-*
    owns_data_models: string[]                      // FKs to data models (modeling phase, parallel to L4a)
    interfaces_provided: string[]                   // FKs to IFACE-*
    interfaces_consumed: string[]                   // FKs to IFACE-*
    estimated_complexity: "low" | "medium" | "high"
    rad_decomposition_status: "leaf" | "decomposed"
  }
  interfaces[]: {
    interface_id: string                            // "IFACE-VENDOR-LOOKUP"
    type: "rest" | "rpc" | "event" | "internal"
    label: string
    contract: string                                // input/output description (free text in v1, formalizable later)
    provider_component_id: string                   // FK
    consumer_component_ids: string[]                // FKs
    serves_capabilities: string[]                   // FKs to CAP-*
  }
}
```

**Concern A validators at L4a recursion boundary**:

1. Every component's `bounded_context_id` resolves.
2. Every component's `supported_capabilities` IDs resolve.
3. Every interface's `provider_component_id` and `consumer_component_ids` resolve.
4. **Capability coverage at every recursion level**: at any RAD level, the union of `supported_capabilities` across the children of a parent component must equal the parent's `supported_capabilities`. No capabilities lost in decomposition; no new capabilities introduced.
5. **Halt rule**: every leaf component (`rad_decomposition_status: "leaf"`) supports exactly one leaf capability from the L3 capability tree. If a leaf component supports multiple leaf capabilities, RAD is required to decompose it further. If a leaf component supports zero leaf capabilities, it is invalid.
6. **Cross-context interfaces are flagged**: any interface where the provider and consumer are in different bounded contexts is recorded and surfaced for review (not blocked, but visible).

**Recursion**: when a component is flagged `estimated_complexity: "high"` and supports more than one leaf capability, the L4a agent decomposes it into sub-components. The decomposition is itself an L4a invocation with a narrower input scope (just the parent component and its supported capabilities). The same validators apply at every recursion level.

**Halt condition**: leaf component supports exactly one leaf capability. This is the deterministic stopping rule for the recursion.

### 5.5 L4b — Tasks (recursive)

**Job**: Take a leaf component from L4a and decompose it into implementation tasks. Each task carries acceptance examples (specification by example). Recursively decompose any task that is too large to be executed by a single LLM agent invocation.

**Input**: One leaf component from L4a, including its supported capability, its interfaces, and its data models.

**Internal framework**: Recursive task decomposition with Specification by Example at the leaves.

**Output schema**:

```ts
L4b_Output {
  tasks[]: {
    task_id: string                                 // "TASK-CONTRACTOR-MATCH-001"
    parent_task_id: string | null
    component_id: string                            // FK to L4a leaf component
    label: string
    description: string
    deliverable_kind: "source_code" | "config" | "schema" | "test" | "other"
    acceptance_examples[]: {                        // Specification by Example
      example_id: string
      given: string                                 // free text or structured precondition
      when: string                                  // free text or structured action
      then: string                                  // free text or structured postcondition
    }
    decomposition_status: "atomic" | "decomposed"
    halt_signal: "single_agent_executable" | "needs_decomposition"
  }
}
```

**Concern A validators at L4b recursion boundary**:

1. Every task's `component_id` resolves.
2. Every task's `parent_task_id` (if non-null) resolves and is in the same component subtree.
3. The task tree is acyclic.
4. **Acceptance example coverage at every level**: union of `acceptance_examples` across the children of a parent task must cover the parent's `acceptance_examples`. Children may add more specific examples but may not drop parent examples.
5. **Halt rule**: a task with `halt_signal: "single_agent_executable"` must have `decomposition_status: "atomic"` and no children. A task with `halt_signal: "needs_decomposition"` must have `decomposition_status: "decomposed"` and at least two children.
6. **Coverage of the component's capability**: union of `acceptance_examples` across all leaf tasks under a leaf component must satisfy the component's supported capability's `acceptance_intent` (this is partially LLM-judged in v1, gradually replaced with deterministic checks).

**Recursion**: a task is initially produced with `halt_signal: "needs_decomposition"` if the agent that produced it judges it too large for a single execution. The next L4b invocation decomposes that task into sub-tasks. The recursion halts when every leaf has `halt_signal: "single_agent_executable"`.

**Halt condition**: every leaf task is single-LLM-agent executable. The halt is *self-declared* by the producing agent at task generation time, then *verified* at execution time: if L4-EXEC attempts to execute an "atomic" task and cannot complete it in one invocation, the task is bounced back to L4b for further decomposition.

### 5.6 L4-EXEC — Execution

**Job**: Take a leaf task from L4b and produce both source code (implementing the task) and test code (verifying the task's acceptance examples).

**Input**: One leaf task from L4b, including its acceptance examples, its parent component's interfaces and data models, and any context from upstream phases needed to write correct code.

**Output**: Source code files and test code files in the project repository, plus a manifest describing what was written.

**Concern A validators at L4-EXEC boundary**:

1. Every leaf task from L4b has produced at least one source code artifact and at least one test code artifact.
2. Every produced test artifact references at least one of the task's acceptance examples (by ID, in a comment or test name).
3. The produced source code passes the project's existing typecheck/lint configuration.
4. The produced source code does not modify files outside its declared scope (e.g., a task scoped to `src/modules/vendor/` does not touch `src/modules/scheduling/`).

**Concern B (V-model) artifacts produced at L4-EXEC**:

Each layer of decomposition produces a corresponding layer of test code. The recursion of the V-model matches the recursion of L4a and L4b:

- **L4b leaf task → unit tests**: test individual functions/classes the executor wrote, one acceptance example per test case (or related group).
- **L4a leaf component → component tests**: test the assembled component (multiple functions composed) against its supported capability's `acceptance_intent`.
- **L4a parent component (every recursion level) → integration tests**: test the assembled sub-components together against the parent's `supported_capabilities`. At deeper recursion levels, integration tests are smaller. At shallower levels, integration tests are larger.
- **L3 capability → capability tests**: test the full set of components implementing a leaf capability against its `acceptance_intent`. These run end-to-end through whatever components are needed.
- **L2 journey → journey acceptance tests**: walk the journey step-by-step against the running system, asserting that each step's `outputs` are produced as the journey describes.
- **L1 stakeholder need → outcome tests** (optional, advanced): assert that the user's `expected_outcome` is achievable from the running system. These are the highest-level black-box tests.

**Test recursion mirrors decomposition recursion**: if L4a recursively decomposes a component three levels deep, there are integration tests at all three levels (innermost integration test covers the deepest sub-component group; outermost integration test covers the whole top component). This is the V-model in its strict recursive form.

**Build/CI integration**: in v1 scope, JanumiCode produces test code that runs in the project's existing test framework (vitest, jest, pytest, etc.). It does not modify the CI pipeline beyond adding test files. Future phases will add CI configuration generation for more sophisticated environments (e.g., generating GitHub Actions workflows, configuring test sharding for large suites). Enterprise-scale CI integration (e.g., the JPMorgan Chase scenario you mentioned) is deferred indefinitely.

- `ENG-DM-` and `DAT-MD-` survive as already-applied renames at the L4a and modeling layers respectively. They are not affected by this design.
- The vocabulary rename is **not applied yet**. It is part of the implementation work that follows approval of this design.

## 7. Two open questions for your decision

These are presented for your decision before any implementation. They are the only structural questions in the chain that I am not unilaterally recommending.

### Open Question 1 — How does the L3 agent propose initial bounded contexts?

Context: when L3 receives the journeys from L2, it needs a strategy for proposing what the bounded contexts are. The two viable strategies are described in §5.3.

- **Path A — Step grouping** (LLM clusters journey steps by affinity)
- **Path B — Glossary first** (LLM extracts vocabulary, clusters vocabulary, each cluster becomes a candidate context)

**My recommendation**: Path B. It is closer to Eric Evans's actual definition of a bounded context, it is more deterministic (vocabulary extraction is closer to a procedure than to a judgment call), and it produces output that the validator can check more rigorously.

**Your call.**

### Open Question 2 — How does the L3 agent name and scope capabilities within a context?

- **Path A — Action-derived** (one capability per journey step)
- **Path B — Need-derived** (capabilities cluster around stakeholder needs)
- **Path C — Hybrid** (need-derived for granularity, with explicit step ID linking for traceability)

**My recommendation**: Path C, already reflected in the L3 schema in §5.3. Both `stakeholder_needs_served` and `contributing_journey_step_ids` are required fields.

**Your call.**

## 8. Phase prompt template skeletons (not final wording)

These are the structural shapes of the system prompts each phase agent will receive. Final wording is deferred to the implementation phase. These are presented to confirm the *structure* matches the design, not to be the final prompts.

### L1 prompt skeleton

```
You are the INTAKE agent. Your job is to take a user's underspecified product or feature
prompt and produce a structured set of stakeholder needs.

You will receive: the user's free-form prompt.

You will produce: a JSON object with the L1_Output schema.

Bloom: be generous in proposing stakeholders and needs. The user will prune later.
       It is better to over-propose than to under-propose.
Constraints:
  - Every stakeholder_need MUST have all three of situation/motivation/expected_outcome.
  - Every need MUST reference an existing stakeholder.
  - Vision statement MUST be a single coherent sentence.
  - Output MUST be valid JSON matching the schema. No prose outside the JSON.
```

### L2 prompt skeleton

```
You are the EXPERIENCE agent. Your job is to take a set of stakeholder needs and produce
concrete user journeys describing how the user satisfies each need.

You will receive: a list of stakeholder_needs from L1.

You will produce: a JSON object with the L2_Output schema.

Constraints:
  - Every journey MUST be linked to at least one stakeholder_need by ID.
  - Every step MUST have a sequence_index, an actor, and a non-empty action.
  - Every journey MUST have exactly one entry step (sequence_index 1) and at least one terminal step (empty next_step_ids).
  - The step graph MUST be acyclic.
  - Do NOT invent backstage processes, support layers, or system internals.
    Only describe what the user does and what the system response is.
  - Output MUST be valid JSON matching the schema. No prose outside the JSON.
```

### L3 prompt skeleton

```
You are the STRUCTURE agent. Your job is to take a set of user journeys and produce
(a) a set of bounded contexts that partition the system's vocabulary and ownership,
and (b) a capability tree where each capability lives in exactly one bounded context.

You will receive: a list of journeys from L2 (with their full step graphs).

You will produce: a JSON object with the L3_Output schema.

Approach:
1. Extract every distinct noun and verb from every journey step.
2. Group nouns and verbs by linguistic affinity. Each group becomes a candidate bounded context.
3. For each bounded context, define its ubiquitous_language (the nouns/verbs that have one specific meaning here).
4. For each stakeholder_need served by the journeys in this bounded context, propose one or more capabilities that satisfy the need.
5. Each capability MUST list its contributing_journey_step_ids. Steps that imply but do not directly invoke a capability still belong to it.
6. Capabilities may be hierarchical, but the hierarchy MUST stay within a single bounded context.

Constraints:
  - Every journey step MUST be referenced by at least one capability's contributing_journey_step_ids.
  - Every capability MUST have a non-empty acceptance_intent.
  - Every capability MUST belong to exactly one bounded context.
  - Capability hierarchies MUST NOT cross bounded context boundaries.
  - Output MUST be valid JSON matching the schema. No prose outside the JSON.
```

### L4a prompt skeleton (initial pass)

```
You are the ARCHITECTURE agent. Your job is to take a capability tree and produce a
component-and-interface decomposition such that every leaf capability is supported by
exactly one leaf component.

You will receive: bounded_contexts and capability_tree from L3.

You will produce: a JSON object with the L4a_Output schema.

Approach:
1. For each bounded context, propose components that own its data and serve its capabilities.
2. For each capability in the capability tree, identify which component(s) support it.
3. For each component, identify the interfaces it provides and consumes.
4. Mark any component as estimated_complexity: high if it supports more than 3 leaf capabilities.
   These will be passed to RAD for further decomposition.
5. Set rad_decomposition_status: leaf for components that are not flagged for further decomposition.

Constraints:
  - Every leaf capability from L3 MUST appear in at least one component's supported_capabilities.
  - At leaf level (post-RAD), every component MUST support exactly one leaf capability.
  - Cross-bounded-context interfaces MUST be explicitly flagged.
  - Output MUST be valid JSON matching the schema.
```

### L4a prompt skeleton (RAD recursion pass)

```
You are the ARCHITECTURE agent in RAD (Recursive Architecture Decomposition) mode.
Your job is to decompose ONE component into sub-components such that the sub-components
collectively cover the parent's supported_capabilities.

You will receive: a single parent component and its full context.

You will produce: a JSON object with the L4a_Output schema, containing only the new sub-components.

Approach:
1. Read the parent's supported_capabilities.
2. Decompose into sub-components such that each sub-component supports a meaningful subset.
3. The union of sub-component supported_capabilities MUST exactly equal the parent's set.
4. Define interfaces between the sub-components if they need to talk to each other.
5. For each sub-component, decide whether it is itself a leaf or needs further decomposition.

Constraints:
  - No sub-component may support a capability the parent does not support.
  - No capability the parent supports may be unsupported in the union of sub-components.
  - Every sub-component MUST have parent_component_id pointing at the parent.
  - Output MUST be valid JSON matching the schema.
```

### L4b prompt skeleton

```
You are the TASK agent. Your job is to take a leaf component from L4a and produce a
recursive task tree that, when executed, will realize the component.

You will receive: one leaf component, its supported capability, its interfaces, and its data models.

You will produce: a JSON object with the L4b_Output schema.

Approach:
1. Read the component's supported capability and its acceptance_intent.
2. Propose tasks that, taken together, will produce the source code and test code for the component.
3. For each task, write acceptance_examples in given/when/then form.
4. Mark each task as either single_agent_executable (atomic) or needs_decomposition.
5. If a task is needs_decomposition, leave its acceptance_examples as a coverage target;
   the next L4b invocation will decompose it into sub-tasks whose examples cover the parent's.

Constraints:
  - Every task MUST have at least one acceptance_example.
  - Every parent task's acceptance_examples MUST be covered by the union of its children's examples.
  - The task tree MUST be acyclic.
  - Output MUST be valid JSON matching the schema.
```

### L4-EXEC prompt skeleton

```
You are the EXECUTION agent. Your job is to take ONE atomic task and produce both
source code (implementing the task) and test code (verifying the task's acceptance examples).

You will receive: one leaf task with its acceptance examples and surrounding context.

You will produce: source files and test files written into the project repository, plus
a manifest describing what was written.

Approach:
1. Read the task's acceptance examples carefully.
2. Write the source code that satisfies all of them.
3. Write a test file with one test per acceptance example. Each test MUST reference its
   acceptance example by example_id in a comment or test name.
4. Run the test file locally if possible. If any tests fail, iterate before returning.
5. If the task turns out to be too large for a single invocation, STOP and signal
   "needs_decomposition" instead of producing partial code.

Constraints:
  - Source code MUST pass the project's typecheck and lint.
  - Source code MUST NOT modify files outside the task's declared scope.
  - Every acceptance example MUST be referenced by at least one test.
  - Output MUST be a JSON manifest listing every file written.
```

These skeletons are intentionally short and constraint-focused. The final prompts will include more context (per-project conventions, code style preferences, framework-specific guidance), but the constraint structure is the load-bearing part and is what determines whether the phase boundary validators can certify the output.
